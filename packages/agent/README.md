# @tokentimer/agent

TokenTimer CertOps Agent: the outbound-only execution-plane process that holds
private keys on the host, talks to the TokenTimer control plane, and deploys
certificates.

Package version tracks the TokenTimer Core release line (`0.11.0` with this PR).

## Distribution model (H10)

**Decision: private package, tarball installer only (not published to the public npm registry).**

| Channel | Status |
| --- | --- |
| Public npm (`npm publish`) | **Not used.** `private: true` remains set. |
| Private/registry npm | Not configured in this repo yet. |
| Release tarball + `install-agent.sh` | **Supported distribution path.** |

Build a verifiable artifact from `packages/agent`:

```sh
pnpm --filter @tokentimer/agent pack:release
# writes dist/tokentimer-agent-<version>.tgz
# and dist/tokentimer-agent-<version>.tgz.sha256
```

Operators verify integrity before install:

```sh
sha256sum -c tokentimer-agent-0.11.0.tgz.sha256
tar -xzf tokentimer-agent-0.11.0.tgz
cd package
sudo ./scripts/install-agent.sh --api-url https://cp.example.com --workspace-id <id> \
  --write-path /etc/letsencrypt --reload-service nginx
```

### Signing / SBOM gap

This repository does not yet ship:

- Sigstore / cosign / GPG signatures for the agent tarball
- An automated CycloneDX/SPDX SBOM publish step for the agent artifact

Checksums (SHA-256) are generated by `pack:release` so operators can verify
bit-for-bit integrity against a trusted checksum channel (release notes,
internal artifact store, etc.). Full signing + SBOM should be added when the
shared release pipeline gains those primitives; do not invent a one-off
signing scheme only for this package.

## Upgrade / rollback

Upgrade (preserves `/opt/tokentimer-agent/state`):

1. Verify the new tarball checksum.
2. Extract and run `sudo ./scripts/install-agent.sh ...` with the same
   `--api-url` / `--workspace-id` / `--write-path` / `--reload-service` values.
3. The installer stages into `app.staging.$$`, swaps atomically, and restarts
   the systemd unit. Credentials and keys in `state/` are left untouched.

Rollback:

1. Keep the previous tarball + `.sha256` from the last known-good release.
2. Re-run install from that tarball (same flags).
3. If the new unit misbehaves before you can reinstall, `systemctl stop
   tokentimer-agent` and restore the previous app tree only if you retained a
   backup; the installer itself removes `app.previous.$$` after a successful
   swap, so retain release tarballs as the rollback source of truth.

Uninstall (keeps state for forensic recovery):

```sh
sudo ./scripts/install-agent.sh --uninstall
# then, only when intentional:
# sudo rm -rf /opt/tokentimer-agent && sudo userdel tokentimer-agent
```
